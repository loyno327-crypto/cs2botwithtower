# Внедрение кликера — план и прогресс

Этот файл — единственный источник правды о том, что уже сделано и что
осталось, чтобы можно было продолжить в новом чате без потери контекста.
Обновляется после каждого этапа.

## Что за фича

Idle/Cookie-Clicker внутри существующего Web App бота (CS2 Case Bot):
игрок тапает по кнопке, получает валюту кликера, прокачивает силу клика
и пассивный доход (кристаллы/час), может обменять кристаллы на монеты
бота. Отдельная вкладка в нижнем меню Web App.

## Ключевые решения (зафиксированы с пользователем)

- Тип: **idle-стиль** — тап + апгрейды + пассивный доход в час (не просто
  тап-на-монету, не завязано на существующую PvE-башню).
- Валюта: **отдельная** от монет бота — "Кристаллы" 🔷 (`config.CLICKER_CURRENCY_NAME`,
  `CLICKER_CURRENCY_ICON`). Обменивается на монеты бота по курсу
  `CLICKER_EXCHANGE_RATE` (сейчас 100 кристаллов = 1 монета).
- Размещение в UI: **новая вкладка в нижнем меню** (`<nav class="tabbar">`),
  наравне с Главная/Игры/Кейсы/Инвентарь/Профиль/Топ — НЕ внутри раздела "Игры".
- Два независимых трека прокачки (без списка отдельных предметов апгрейда,
  просто уровень с растущей ценой):
  - **Сила клика** (`click_level`) — сколько кристаллов даёт 1 тап.
  - **Автосборщик** (`passive_level`) — сколько кристаллов начисляется в час,
    даже когда игрок не в приложении (до `CLICKER_MAX_OFFLINE_HOURS` = 12 часов
    простоя разом, дальше не копится, чтобы нельзя было получить бонус за
    "месяц отсутствия").
  - Цена следующего уровня = `base_cost * growth ** текущий_уровень`
    (экспоненциальный рост, отдельные base_cost/growth для каждого трека
    в config.py).
- Античит: тапы шлются на сервер по одному, сервер проверяет минимальный
  интервал между тапами (`CLICKER_MIN_TAP_INTERVAL_MS` = 80мс) — защита от
  накрутки скриптом в обход интерфейса.

## Этапы

### ✅ Этап 1 — База данных и конфиг (СДЕЛАНО)

- `config.py`: добавлена секция "Кликер" в конец файла — название/иконка
  валюты, базовые значения силы клика и пассивного дохода, стоимость и
  рост цены апгрейдов, курс обмена, кап оффлайн-часов, антиспам-интервал.
- `database.py`:
  - новая таблица `clicker_state` (создаётся в `init_db()`): currency,
    click_level, passive_level, total_taps, total_earned,
    total_exchanged_currency, total_exchanged_coins, last_collected_at,
    last_tap_at.
  - публичные функции (все сами создают строку игрока при первом обращении
    и сами применяют накопившийся пассивный доход перед своим действием —
    снаружи ничего отдельно вызывать не нужно):
    - `get_clicker_state(user_id) -> dict` — текущее состояние +
      вычисленные click_power/passive_income/стоимости следующих апгрейдов.
    - `clicker_tap(user_id) -> dict` — один тап, начисляет click_power,
      возвращает state + `gained`.
    - `clicker_get_last_tap_at(user_id) -> str|None` — только чтение метки
      последнего тапа, для проверки антиспама (её должен использовать
      Этап 2 в webserver.py **до** вызова `clicker_tap`).
    - `clicker_buy_upgrade(user_id, track)` — track: `"click"` или
      `"passive"`. Возвращает `{success, error, state}`.
    - `clicker_exchange(user_id)` — обменивает ВСЕ доступные кристаллы на
      монеты по курсу (остаток меньше курса остаётся в кристаллах),
      начисляет монеты через `add_balance(..., reason="clicker_exchange")`.
  - Все покупки/обмены пишутся в `event_log` через `log_event` с
    `event_type` = `"clicker_upgrade"` / `"clicker_exchange"`.
  - Протестировано вручную (тапы, покупка апгрейдов, обмен, накопление
    пассивного дохода за прошедшее время) — работает корректно.

### ✅ Этап 2 — API на сервере (СДЕЛАНО)

`webserver.py`: добавлен импорт `from datetime import datetime` (нужен для
проверки антиспама тапов) и 4 хендлера + регистрация роутов, тем же стилем,
что и остальные `handle_*` (авторизация через `_authenticate`/`_unauthorized`):

- `GET /api/clicker/state` → `{state, currency_name, currency_icon, exchange_rate}`.
  `state` — то, что возвращает `db.get_clicker_state` (currency, click_level,
  passive_level, click_power, passive_income, click_upgrade_cost,
  passive_upgrade_cost, total_taps, total_earned, total_exchanged_currency,
  total_exchanged_coins).
- `POST /api/clicker/tap` → перед начислением сам проверяет интервал с
  прошлого тапа через `db.clicker_get_last_tap_at` +
  `config.CLICKER_MIN_TAP_INTERVAL_MS`; если тапнули слишком быстро — 429
  `{"error": "too_fast"}` БЕЗ начисления и без обращения к `clicker_tap`
  (защита от накрутки скриптом). Иначе → `{state}` (state включает `gained`
  — сколько кристаллов дал именно этот тап).
- `POST /api/clicker/upgrade` → тело `{"track": "click"|"passive"}` →
  `{success, error, state}` (400, если трек невалиден или не хватает
  валюты — `error: "invalid_track"` / `"not_enough_currency"`).
- `POST /api/clicker/exchange` → без тела, меняет все кристаллы →
  `{success, error, coins_gained, state}` (400 + `error: "not_enough_currency"`,
  если кристаллов меньше курса обмена).

Живой интеграционный тест (aiohttp TestClient) прогнать в этой песочнице не
удалось — нет сетевого доступа, чтобы поставить `aiohttp` (он не в
`requirements.txt` напрямую, тянется транзитивно через `aiogram`, так что на
реальном хостинге он есть). Код перепроверен вручную построчно, логика 1:1
повторяет уже протестированные в Этапе 1 функции `database.py`. **На реальном
деплое эндпоинты стоит один раз проверить руками** (curl/Postman или прямо
из будущего фронтенда Этапа 3) перед тем как считать фичу законченной.

### ✅ Этап 3 — Фронтенд (СДЕЛАНО)

`webapp/index.html`:

- **Нижнее меню**: добавлена кнопка `data-tab="clicker"` (🔷 Кликер) между
  "Игры" и "Кейсы". На Главной экране в `.quick-grid` тоже добавлена плитка
  быстрого перехода.
- **Разметка**: новый `<div class="view" id="view-clicker">` — большая
  круглая кнопка-тапалка с CSS pulse-анимацией на каждый тап, счётчик
  кристаллов и подпись валюты (название/иконка приходят с сервера, не
  зашиты в HTML), строка с текущей силой клика / доходом в час, слой для
  всплывающих "+N" (`clicker-float-layer`), блок с двумя карточками
  апгрейдов (сила клика / автосборщик — каждая с кнопкой покупки и текущей
  ценой следующего уровня) и кнопка "Обменять на монеты" (курс — тоже с
  сервера).
- **CSS**: новая секция стилей `/* ---- Кликер ---- */` перед `nav.tabbar`,
  использует только существующие CSS-переменные (`--amber`, `--panel`,
  `--line` и т.д.) — отдельной цветовой схемы не заводили.
- **JS**: блок `// ---------- Кликер ----------` перед секцией навигации
  по вкладкам, зарегистрирован в `tabRenderers.clicker = initClickerView`.
  Логика:
  - `initClickerView()` — грузит `/api/clicker/state`, сохраняет
    `clickerMeta` (название/иконка валюты, курс обмена) и `clickerState`.
  - `clickerTap()` — на каждый клик всегда проигрывает pulse-анимацию
    (отклик на нажатие всегда мгновенный), но реальный запрос на сервер
    шлёт не чаще `CLICKER_TAP_MIN_INTERVAL_MS` = 80мс (захардкожено в JS,
    **должно совпадать** с `config.CLICKER_MIN_TAP_INTERVAL_MS` — если
    поменяешь одно, поменяй и другое). Сервер всё равно главный источник
    истины: 429 от него просто тихо игнорируется, `currency` на экране
    обновляется только из ответа сервера (никакого оптимистичного счёта
    "на глаз" — цифры всегда настоящие).
  - `buyClickerUpgrade(track)` / `clickerExchange()` — POST на
    соответствующие эндпоинты, показывают текст ошибки в `#clicker-error`
    (используется существующий класс `.crash-error`, отдельный не заводили).
    После успешного обмена дополнительно дёргается `loadProfile()`, чтобы
    баланс монет в шапке обновился сразу.
- Проверено: `node --check` на извлечённом `<script>` (синтаксис ОК),
  баланс открывающих/закрывающих `<div>` (204/204), уникальность всех
  новых `id`. Живого рендера в браузере не делал (нет соответствующего
  инструмента в песочнице) — **стоит открыть Web App в Telegram и
  визуально проверить вкладку** перед тем как считать её законченной.

### ✅ Этап 4 — Полировка (СДЕЛАНО)

- Всплывающие "+N" при тапе и вибрация на тап/успешные действия —
  оказались уже реализованы в Этапе 3 (`spawnClickerFloat`,
  `tg.HapticFeedback`), отдельно ничего не добавлял.
- **Уведомление об оффлайн-доходе**: `database.py` —
  `_apply_clicker_passive_income` теперь возвращает
  `(row, начислено_за_этот_вызов)` вместо просто `row` (все вызывающие
  функции обновлены на распаковку кортежа). `get_clicker_state` кладёт
  это значение в `offline_collected` в ответе — 0, если автосборщик не
  куплен или доход только что забирали. `webapp/index.html`:
  `initClickerView` при `offline_collected > 0` показывает баннер
  "Пока тебя не было, автосборщик накопил +N 🔷" над сценой кликера
  (`#clicker-offline-banner`, стили `.clicker-offline-banner`) — баннер
  разовый, при следующем открытии вкладки не показывается заново, если
  показывать нечего.
- **Статистика кликера в "Профиль"**: `webserver.py` — `handle_stats`
  дополнительно дёргает `db.get_clicker_state` и добавляет в ответ
  `clicker_currency`, `clicker_click_level`, `clicker_passive_level`,
  `clicker_total_taps`, `clicker_total_earned`,
  `clicker_total_exchanged_currency`, `clicker_total_exchanged_coins`.
  `webapp/index.html` — `renderProfileTab` добавляет 4 строки в
  существующий `.profile-stat-list` (тапов всего, заработано кристаллов,
  обменяно на монеты, уровни силы клика/автосборщика).
- Проверено: `python3 -m py_compile database.py webserver.py` — ОК;
  `node --check` на извлечённом `<script>` — ОК; баланс `<div>` (206/206);
  новый id `clicker-offline-banner` уникален (пара других дублей id в
  файле — `slots-bet-grid`/`crash-again-btn` — уже были раньше, не
  относятся к этой правке, это разные ветки рендера одной и той же
  разметки, не одновременно в DOM). Живого запуска бота и визуальной
  проверки в Telegram не делал — **стоит открыть Web App и проверить
  вкладки "Кликер" и "Профиль" глазами** перед тем как считать фичу
  полностью законченной.
